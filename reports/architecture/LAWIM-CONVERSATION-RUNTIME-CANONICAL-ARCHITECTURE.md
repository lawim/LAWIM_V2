# LAWIM V2 — Canonical Conversation Runtime Architecture

**Date :** 2026-07-20
**HEAD :** `b2fe5e34`

## Single Canonical Runtime

`ConversationStateEngine.process_turn()` is the sole entry point for all conversational response generation.

```python
# communication/service.py:628
result = self.conversation_state_engine.process_turn(
    actor_id=actor_id,
    channel=channel,
    external_conversation_id=conversation_key,
    message=raw_text,
    language=language,
)
```

## States and Data Types

| Type | Location | Role |
|------|----------|------|
| `ConversationState` | `conversation/state/state.py:21` | Full conversation snapshot: intent, known_slots, missing_slots, language, qualification_status, handover_status |
| `ConversationTurnDecision` | `conversation/state/state.py:108` | Decision produced during a turn: intent, next_action, next_question_key, handover_required, allowed/forbidden content |
| `ResponsePlan` | `conversation/state/state.py:85` | Mandatory plan before any provider call: response_type, next_question, maximum_questions, forbidden_content, handover fields |
| `ConversationTurnContext` | `conversation/state/state.py:9` | Input context for a turn: actor, channel, message, language, previous state |

**Note:** `ConversationTurnResult` does not exist as a dataclass — `process_turn()` returns a plain `dict[str, Any]` containing `state`, `response`, `response_plan`, `handover_required`, and `wizard_completed`.

## ResponsePlan is Mandatory

Every response generated by the system must have a `ResponsePlan` before any provider call. This is verified by:

- **Architecture contract test** `test_response_plan_exists_before_provider_call` — creates a real `ConversationStateEngine` and verifies that `_generate_ai_reply("Bonjour", "whatsapp", "test-key")` returns a valid response through the engine's ResponsePlan.
- **Architecture contract test** `test_turn_decision_exists_before_response_plan` — verifies `process_turn()` returns a `response_plan` key in its result dict.
- **Fallback path** creates a synthetic `ResponsePlan(maximum_questions=1)` before calling `ConversationResponseValidator.validate()`.

## ResponseValidator is Active

`ConversationResponseValidator` (at `conversation/state/validator.py:64`) checks every response:

1. **Forbidden content detection** (`detect_forbidden_content`, line 43):
   - Neutral assistant phrases ("neutral assistant", "I cannot make business decisions")
   - External platform referrals ("Jumia", "SeLoger", "Leboncoin", "Facebook", "Lamudi")
   - Unrequested translation ("french for", "in english", "français signifie")
   - Unrequested grammar correction ("correct spelling is", "the correct phrasing")

2. **Question count enforcement**: If `plan.maximum_questions > 0` and the response contains more questions than allowed, it is replaced with `plan.next_question_text` (status `REPAIR`).

3. **If REPAIR**: The repaired text replaces the original response.

## All Channels Go Through CommunicationService._generate_ai_reply

Both WhatsApp and Telegram webhook handlers call `_generate_ai_reply()` as their single point of response generation:

```
process_green_api_webhook  →  _generate_ai_reply  →  ConversationStateEngine
process_telegram_webhook   →  _generate_ai_reply  →  ConversationStateEngine
```

The Web V3 channel (`_handle_v3_conversation_message_post`) currently only persists messages and does NOT call `_generate_ai_reply`. It returns `{status: "received"}` with no response.

The CRM channel (`crm.send_whatsapp`) is an admin-only direct send that bypasses the runtime entirely.
