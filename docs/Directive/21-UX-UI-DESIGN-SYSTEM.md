# LAWIM

# 21-UX-UI-DESIGN-SYSTEM.md

# Système de design UX/UI officiel

Version 1.0

---

# CHAPITRE 1 — OBJECTIF

Le présent document définit l'identité visuelle et les règles de conception de LAWIM.

Il constitue la référence unique pour :

* l'identité visuelle ;
* les couleurs ;
* la typographie ;
* les composants ;
* les formulaires ;
* les dashboards ;
* les notifications visibles ;
* les écrans de paiement.

---

# CHAPITRE 2 — PRINCIPES DE CONCEPTION

Le design de LAWIM doit être :

* lisible ;
* rassurant ;
* accueillant ;
* clair ;
* cohérent entre Web et Mobile ;
* légèrement ludique sans nuire au sérieux métier.

Les composants doivent servir l'action, pas distraire.

---

# CHAPITRE 3 — IDENTITÉ VISUELLE

L'identité visuelle doit garantir :

* une signature LAWIM identifiable ;
* des contrastes lisibles ;
* une hiérarchie claire ;
* des repères visuels constants ;
* une bonne perception sur petit écran.

Les couleurs doivent soutenir l'information, pas la masquer.

---

# CHAPITRE 4 — TYPOGRAPHIE ET ICONOGRAPHIE

La typographie doit être :

* moderne ;
* lisible ;
* hiérarchisée ;
* adaptée aux écrans mobiles et desktop.

Les icônes doivent :

* être cohérentes ;
* être explicites ;
* être réutilisables ;
* renforcer la compréhension rapide.

---

# CHAPITRE 5 — COMPOSANTS

Les composants officiels comprennent notamment :

* boutons ;
* cartes ;
* listes ;
* badges ;
* formulaires ;
* champs de saisie ;
* tableaux ;
* statuts ;
* modales ;
* alertes.

Chaque composant doit être documenté et réutilisable.

---

# CHAPITRE 6 — DASHBOARDS ET FORMULAIRES

Les dashboards doivent :

* mettre en avant les actions critiques ;
* réduire le bruit visuel ;
* garder les KPI lisibles ;
* hiérarchiser les blocages.

Les formulaires doivent :

* éviter les champs inutiles ;
* guider la saisie ;
* signaler les erreurs clairement ;
* respecter les référentiels métier.

---

# CHAPITRE 7 — ACCESSIBILITÉ ET RESPONSIVE

Le design doit rester :

* accessible ;
* responsive ;
* compatible avec les petits écrans ;
* lisible avec des contrastes raisonnables ;
* utilisable au clavier et au toucher.

---

# CHAPITRE 8 — ANIMATIONS ET RETOURS VISUELS

Les animations doivent être :

* utiles ;
* courtes ;
* cohérentes ;
* non intrusives.

Les changements d'état doivent être visibles sans ambiguïté.

---

# CHAPITRE 9 — NOTIFICATIONS ET PAIEMENT

Les notifications importantes doivent être visibles, à la manière d'un système d'alerte fort.

Les écrans de paiement et de confirmation doivent être :

* clairs ;
* rassurants ;
* explicites ;
* cohérents avec la validation backend.

---

# CHAPITRE 10 — TESTS

Les tests du design system doivent vérifier :

* cohérence Web/Mobile ;
* lisibilité ;
* accessibilité ;
* états de chargement ;
* états d'erreur ;
* écrans de paiement ;
* notifications critiques.

---

# CHAPITRE 11 — SUPPORT MULTILINGUE

Le système de design doit garantir :

* des libellés gérés par clés de traduction ;
* des composants capables d'absorber l'expansion du texte ;
* des formats régionaux cohérents pour les dates, nombres et devises ;
* l'absence de chaîne codée en dur dans les écrans produits ;
* une cohérence visuelle quel que soit le sens d'affichage futur.

Le design system s'appuie sur 30-I18N-L10N-REFERENCE.md et 30B-TRANSLATION-REFERENCE.md.

---

# CHAPITRE 12 — TRACKING MARKETING ET AFFICHAGE

Les écrans existants peuvent afficher :

* les campagnes ;
* les publications ;
* les performances par canal ;
* les codes de tracking ;
* les statistiques de conversion ;
* les taux de clic ;
* les attributions.

Le design system doit permettre d'ajouter ces blocs dans les dashboards existants sans créer de nouveau dashboard.

---

# CHAPITRE 13 — OBJECTIF FINAL

Le système de design doit permettre à LAWIM d'offrir une expérience élégante, compréhensible et cohérente sur tous les supports.
